# Configuration reference

> Part of the Cairn reference docs. The section numbers below are stable identifiers used throughout the code and docs; see the index in [`CLAUDE.md`](./CLAUDE.md) and [`../CLAUDE.md`](../CLAUDE.md).

## 28. Configuration reference

### 28.1 Surface and conventions

Server configuration comes entirely from **environment variables** in the `CAIRN_*` namespace, overlaid on the built-in defaults and validated on load so that an invalid configuration fails fast with a clear message rather than at first use. Cairn deliberately has **no configuration file and no server command-line flags**: a single, explicit environment surface keeps a deployment reproducible — identical whether run directly on a host or in a container — and avoids the precedence ambiguity of layering flags over a file over the environment. (The CLI subcommands accept their own flags, as any command-line tool does; this constraint concerns the server's own configuration, not the CLI.) The table below names settings logically; each maps to a `CAIRN_<SETTING>` variable. The settings continue the operational vocabulary an operator coming from a comparable server would expect. Note that **per-bucket replication targets and rules are primarily stored S3 resource state** — set through the management API, the UI, or the CLI (Section 20), with destination credentials sealed at rest under the master key — while a default or fallback target can still be configured through the `CAIRN_REPLICATION_*` environment variables and `CAIRN_REPLICATION_TARGETS`, used for any source bucket that has no stored target.

### 28.2 Settings

| Setting | Variable | Default | Meaning |
|---|---|---|---|
| S3 API listen address | `CAIRN_LISTEN_ADDR` | `0.0.0.0:7373` | Where the S3 data-plane listener binds: the S3 protocol, the signed public-read share URLs, and the liveness, readiness, and metrics endpoints. |
| Web-UI listen address | `CAIRN_UI_ADDR` | `0.0.0.0:7374` | Where the web console, the management API, and the S3 data plane the console drives are served at the root path; set it empty, or `off`/`none`/`disabled`, to run headless with no UI listener. |
| Metadata backend | `CAIRN_META_BACKEND` | `sqlite` | Which engine drives the metadata store: `sqlite` (the bundled-C rusqlite store), `libsql` (the async embedded driver), or `turso` (the pure-Rust SQLite rewrite). |
| TLS certificate and key paths | `CAIRN_TLS_CERT_PATH` / `CAIRN_TLS_KEY_PATH` | unset (plaintext) | Enable built-in TLS when both are set; otherwise serve plaintext behind a proxy. |
| Database path | `CAIRN_DB_PATH` | under the data root | Location of the SQLite metadata file. |
| Data directory | `CAIRN_DATA_DIR` | a data root | Root of the staging and per-bucket blob directories; must share a filesystem with the database. |
| Region | `CAIRN_REGION` | `us-east-1` | The region label returned by the location operation and used in SigV4 scope checks. |
| Public base URL | `CAIRN_PUBLIC_BASE_URL` | unset | External base URL used when generating URLs behind ingress. |
| Virtual-host base domain | `CAIRN_S3_DOMAIN` | unset | The base domain for virtual-host-style addressing (`<bucket>.<domain>`); unset serves path-style only. |
| Metadata cache budget | `CAIRN_META_CACHE_BYTES` | 64 MiB | Byte budget for the metadata and configuration cache; zero disables it. |
| Maximum object size | `CAIRN_MAX_OBJECT_SIZE` | a large ceiling | Hard per-object size limit. |
| Write-ahead-log checkpoint interval | `CAIRN_WAL_CHECKPOINT_INTERVAL_SECS` | on the order of a minute | Cadence of the truncating checkpoint. |
| Write-ahead-log checkpoint size threshold | `CAIRN_WAL_CHECKPOINT_SIZE_BYTES` | 64 MiB | Trigger a truncating checkpoint between interval ticks once the log grows past this; zero leaves only the interval. |
| Multipart session lifetime and sweep interval | `CAIRN_MULTIPART_UPLOAD_LIFETIME_SECS` / `CAIRN_MULTIPART_SWEEP_INTERVAL_SECS` | a day and an hour | When idle uploads become stale and how often the sweeper runs. |
| Lifecycle scan interval | `CAIRN_LIFECYCLE_INTERVAL_SECS` | on the order of hours | How often the lifecycle scanner runs. |
| Webhook delivery interval | `CAIRN_WEBHOOK_INTERVAL_SECS` | `15` | How often the event-notification worker drains the webhook delivery outbox to the configured per-bucket endpoints. The claim is a cheap indexed query, so the loop is a no-op for buckets without notifications. Must be positive. |
| Webhook outbox retention | `CAIRN_EVENTS_OUTBOX_RETENTION_SECS` | `86400` | Retention for terminally-failed webhook-delivery (`events_outbox`) rows, so a misconfigured or decommissioned sink cannot grow the metadata DB without bound; pending/claimed entries (outstanding work) are never pruned and delivered entries are removed on delivery. |
| Integrity scrub interval | `CAIRN_SCRUB_INTERVAL_SECS` | `0` (**off** — no scrub runs until you set this) | How often the background scrub re-reads every stored blob and verifies it against the recorded ETag, turning silent bit-rot into a `cairn_scrub_corruption_total` event. Opt-in and I/O-heavy (it reads every blob). It covers **encrypted objects too** — transparent at-rest, SSE-S3 and SSE-KMS versions are re-read through their own unsealed DEK, so a `CAIRN_ENCRYPT_AT_REST` node is scrubbed exactly like a plaintext one — as well as compressed ones. What it cannot verify is counted, never dropped: `cairn_scrub_skipped_total{reason}` reports versions whose data key is off the ring right now (a rotation window — skipped, retried next pass, never reported as corruption) and multipart objects whose composite `{md5}-{n}` ETag is not a whole-object digest (read and authenticated, but not hash-compared). `scanned + skipped` accounts for every version a pass walked. Because it now re-reads and **decrypts** every version, a pass on an encrypted store is CPU/I/O-heavy in proportion to stored bytes (no rate limiter; one blob-read permit at a time, so live GETs are not starved) — schedule for quiet periods (Sections 26.4, 8.6). |
| Replication interval | `CAIRN_REPLICATION_INTERVAL_SECS` | tens of seconds | How often the replication worker drains the outbox. |
| Request metrics | `CAIRN_REQUEST_METRICS_ENABLED` / `_FLUSH_SECS` / `_BUCKET_SECS` / `_RETENTION_DAYS` | on, 15s flush, 60s window, 31-day retention | Whether the usage-analytics subsystem accumulates request counts, how often the in-memory aggregator flushes and prunes, the rollup window granularity, and how long history is kept (Section 26.5). |
| AWS-STS wire surface | `CAIRN_STS_ENABLED` | on | Whether the AWS-STS minting protocol (`AssumeRole` + `GetSessionToken`) is served on the S3 data-plane port as a form `POST /` returning AWS-STS XML (Section 14), so the AWS SDK default credential-provider chain and Terraform's `assume_role{}` obtain temporary credentials with zero config. Set `false` to shrink the attack surface (a form `POST /` then falls through to normal S3 routing); the management-plane mint (`POST /api/v1/credentials/temporary`) is unaffected either way. |
| At-rest encryption | `CAIRN_ENCRYPT_AT_REST` | off | When on, every committed object is transparently encrypted at rest even if the client sent no SSE header and the bucket has no default encryption: a per-object DEK is minted, sealed under the master-key ring, and the blob is stored encrypted. The object advertises **no** `x-amz-server-side-encryption` — an operator storage property, not an SSE contract the client asked for (the `AtRest` descriptor mode records this). An explicit client SSE header or a bucket default still takes priority and IS advertised. A confidentiality/throughput trade: an encrypted object can engage neither the sendfile zero-copy nor the small-object whole-read GET fast path, so it stays opt-in (Section 27). |
| SSE-KMS key-id allow-list | `CAIRN_KMS_KEY_IDS` | unset (accept-any) | A comma-separated list of key ids accepted on an `x-amz-server-side-encryption: aws:kms` write or a bucket KMS default; blank entries are dropped and a list with no non-blank entries collapses to accept-any. **Unset accepts ANY key id.** The key id is a *label*, not cryptographic isolation — every DEK is sealed under the same node master-key ring regardless of the id — and this list gates **writes only**: removing an id does not lock existing objects, whose reads still unwrap under the master key. Set it to constrain which ids clients may name (Section 27). |
| Metadata durability + throughput | `CAIRN_META_SYNCHRONOUS` / `_GROUP_COMMIT_LINGER_MICROS` / `_READ_POOL_SIZE` / `_CACHE_BYTES_PER_CONN` / `_CACHE_TOTAL_BUDGET_BYTES` / `_MMAP_BYTES` | `full`, 0, `max(8,cores)`, 64 MiB, 2 GiB, 256 MiB | Tune the metadata store (Section 30): write durability (`full`/`normal`), the group-commit linger window (≤1 ms), the WAL read-pool size (≤64), the page cache per connection, the hard total-cache budget (startup refuses a pool×cache combination over this), and the mmap size. **Durability is safe by default: `full` (WAL + `PRAGMA synchronous=FULL`) means an acknowledged write survives power loss.** Set `CAIRN_META_SYNCHRONOUS=normal` for an opt-in throughput mode (≈1.7× writer throughput, no per-commit fsync) that never corrupts but may lose the last few acknowledged writes on power loss. The background checkpointer is the sole WAL checkpointer. |
| Authentication cache TTL | `CAIRN_AUTH_CACHE_TTL_SECS` | 30 s | Time-to-live for the credential + parsed-policy cache (Section 30.4); a user-identity mutation invalidates it immediately via a shared epoch, so the TTL only bounds staleness for untouched entries. Capped at 1 hour; zero disables the cache. |
| Runtime thread sizing | `CAIRN_RUNTIME_WORKER_THREADS` / `_MAX_BLOCKING_THREADS` | auto (CPU count) / auto (`max(512, blob_io_pool + read_pool + 64)`) | Tokio compute and blocking-pool sizing (Section 30.4). `0` auto-derives; an explicit blocking cap is validated to stay at or above the floor the WAL read pool plus blob I/O pool require, so neither starves. |
| Blob I/O permit pools | `CAIRN_BLOB_IO_POOL_SIZE` / `CAIRN_BLOB_IO_READ_POOL_SIZE` | 64 / 64 | Separate permit pools bounding concurrent blob write/assemble and blob read transfers (Section 7.4); each transfer holds one permit for its file I/O, so a flood of large or slow transfers cannot exhaust the runtime's blocking pool. Keeping reads distinct — a read holds its permit for the whole client-paced download — means slow readers starve only other reads, never writes. |
| Sendfile GET size floor | `CAIRN_FASTIO_MIN_BYTES` | 256 KiB | Minimum response size for the experimental `sendfile` GET fast path, consulted only in a `fast-io` build. A GET whose body is below this floor falls back to the normal streamed path, since the per-request sendfile overhead can outweigh the zero-copy saving on a tiny body; `0` disables the floor. No effect without `fast-io`. |
| Metadata shards | `CAIRN_META_SHARDS` | 1 | Partition the metadata across N databases by bucket name so disjoint buckets commit through N parallel single-writers (Section 30.4). `1` is the unsharded store; `>1` is sqlite-only, capped at 64, and is a deployment-time decision fixed at first init (user-quota becomes eventually-consistent). |
| Default replication target | `CAIRN_REPLICATION_ENDPOINT` / `_DEST_BUCKET` / `_ACCESS_KEY` / `_SECRET` / `_REGION` | unset | A single default destination shipped to for any source bucket without a stored target (Section 20). |
| Named replication targets | `CAIRN_REPLICATION_TARGETS` | unset | A JSON array of named destinations; a source bucket is routed to the matching named target. |
| Replication drain tuning | `CAIRN_REPLICATION_BATCH_SIZE` / `_WORKER_CONCURRENCY` / `_MAX_ATTEMPTS` / `_BASE_BACKOFF_SECS` / `_MAX_BACKOFF_SECS` / `_RETENTION_SECS` | 64, 4, 8, 5 s, 900 s, 86 400 s | How many due outbox entries a drain pass claims, how many worker tasks drain concurrently (per-key, per-target ordering is preserved regardless of pool size), the max delivery attempts before a retryable failure becomes terminal, the base and ceiling of the exponential retry backoff (max must be ≥ base), and how long terminal (completed/failed) outbox rows are retained before reclaiming — pending/claimed rows are never pruned (Section 20). |
| Allow plaintext SSE bodies over `http://` replication | `CAIRN_REPLICATION_ALLOW_PLAINTEXT_SSE_OVER_HTTP` | off (refuse) | Replication ships the **decrypted** body: the engine unseals each version's data key and reads the plaintext, because shipping the stored ciphertext would leave the mirror holding bytes no destination can decrypt. When the destination endpoint is plaintext `http://`, that puts data the client explicitly asked to have encrypted (SSE-S3 or SSE-KMS) onto an unauthenticated link, so by default such an object is **refused per object** and rescheduled as *unavailable* — no attempt budget is consumed, and it ships by itself the moment the endpoint becomes `https://` or this is set. Objects encrypted only by transparent at-rest encryption (`CAIRN_ENCRYPT_AT_REST`, an operator storage property rather than a client contract) and unencrypted objects are unaffected and replicate over `http://` as before. Set `true` only where the link is trusted by other means (a private VLAN, a VPN, a service mesh terminating TLS); it logs a loud startup warning. Startup refuses the knob when the only configured endpoint is `https://` and no target list is set, so it can never sit inert. |
| Encrypted-suspect replication audit cutoff | `CAIRN_REPLICATION_AUDIT_BEFORE` | unset (the audit does not run) | The cutoff for the background encrypted-suspect replication audit (Section 20.5), as an RFC-3339 timestamp (`2026-07-23T10:00:00Z`) or whole epoch seconds. Set it to the moment this node was upgraded past the SSE replication defect. **Unset means the loop never runs**: no version walk, no gauges, no warning, zero cost. That is deliberate rather than conservative — only versions written by the *pre-fix* binary can be damaged, and a version encrypted and replicated after the fix is equally encrypted and equally `completed`, so without a cutoff the count includes every healthy encrypted replica, sits permanently non-zero on any node using SSE with replication, and can never reach zero. A permanently-firing alarm shipped as a default is worse than no alarm, so the operator supplies the one fact the node cannot know. With it set, `cairn_replication_encrypted_repair_pending_versions` genuinely converges to zero and `cairn_replication_encrypted_suspect_versions` converges to `cairn_replication_encrypted_non_current_suspect_versions` (the population no repair command can reach), so alerting on the pending series, or on the suspect series minus that floor, is meaningful. An unparseable value fails startup. The one-off `cairn replication audit --before` takes the same value and does not need this knob. |
| Import worker count | `CAIRN_IMPORT_DEFAULT_WORKERS` / `CAIRN_IMPORT_MAX_WORKERS` | 8 / 32 | Default and hard-cap on the per-job object-copy concurrency for an S3 import (Section 27); a job requesting more is clamped, and a job requesting 0 uses the default. `default ≤ max` is validated. |
| Import global in-flight cap | `CAIRN_IMPORT_GLOBAL_MAX_INFLIGHT` | 24 | The authoritative ceiling on total in-flight object copies across an import job. Held **below the blob-I/O permit pool (64)** so a bulk import cannot starve live GET/PUT; validated `1..=63`. |
| Import poll interval and retention | `CAIRN_IMPORT_POLL_INTERVAL_SECS` / `CAIRN_IMPORT_RETENTION_SECS` | 30 / 7 days | How often the import worker wakes to claim a pending/orphaned job, and how long finished (`completed`/`failed`/`cancelled`) job rows are kept before pruning. Both must be positive. |
| Root administrator credentials | `CAIRN_ROOT_ACCESS_KEY` / `CAIRN_ROOT_SECRET_KEY` | a well-known default | The access key and secret of an administrator ensured on every startup; override in production. |
| Master key | `CAIRN_MASTER_KEY` | required when any encrypted secret exists | The authenticated-encryption key for secrets at rest, supplied out of band. |
| Master-key ring and active id | `CAIRN_MASTER_KEY_RING` / `CAIRN_MASTER_KEY_ACTIVE_ID` | unset | A JSON array of `{"id","key"}` entries that replaces `CAIRN_MASTER_KEY` for rotation: new secrets seal under the active key id and old ids stay available to open existing data. `CAIRN_MASTER_KEY_ACTIVE_ID` names which ring id new seals use, defaulting to the highest id (rotate forward only — the active id must be the newest). Leave both unset for a single-key deployment. |
| Master-key re-wrap and counter sync | `CAIRN_KEY_REWRAP_INTERVAL_SECS` / `CAIRN_KEY_COUNTER_SYNC_SECS` | 300 / 60 | How often the re-wrap worker re-seals secrets onto the active master key, and how often the active key's seal counter is flushed to durable state; `0` disables either. sqlite backend only. |
| Concurrency limit and request timeout | `CAIRN_CONCURRENCY_LIMIT` / `CAIRN_REQUEST_TIMEOUT_SECS` | bounded defaults | Maximum in-flight requests and per-request timeout. |
| Maximum connections per listener | `CAIRN_MAX_CONNECTIONS` | `8192` | Cap on concurrent TCP connections accepted per listener; a socket past the cap is dropped immediately (counted as `cairn_connections_rejected_total`), so a flood of idle or slow sockets cannot exhaust file descriptors and memory ahead of the per-request concurrency limiter. |
| Request-head read timeout | `CAIRN_HEADER_READ_TIMEOUT_SECS` | `15` | Maximum time a connection may take to send its complete request head before it is dropped, bounding a slowloris that dribbles or never finishes the request line and headers (the per-request timeout only starts once the head is fully parsed). Applies to both listeners. |
| Development authentication bypass | `CAIRN_DEV_AUTH` | off | Enables the loopback-only development administrator (debug builds only). |
| Allow insecure defaults on a public bind | `CAIRN_ALLOW_INSECURE` | off | `serve` and `validate-config` **refuse to bind a non-loopback `CAIRN_LISTEN_ADDR`** while the built-in development master key (no `CAIRN_MASTER_KEY`) or the default root secret is in use, so a hurried deploy cannot come up fully functional yet fully insecure. Set `true` only on a trusted/closed network where those defaults are acceptable (a demo or internal rig); production should instead set a real `CAIRN_MASTER_KEY` and `CAIRN_ROOT_SECRET_KEY`. Loopback binds are always allowed. |
| Allow internal outbound endpoints (SSRF guard) | `CAIRN_ALLOW_INTERNAL_ENDPOINTS` | off | The SSRF guard refuses an operator-configured outbound endpoint (replication target, webhook receiver, S3 import source) that resolves to a loopback, private (RFC1918), link-local (incl. the cloud-metadata `169.254.169.254`), ULA, or unspecified address — at connect time, so it also defeats DNS rebinding. Set `true` only when Cairn must reach storage on a private network (e.g. an on-prem MinIO on `10.x`); it emits a loud startup warning. See Section 27. |
| Log level and format | `CAIRN_LOG_LEVEL` / `CAIRN_LOG_FORMAT` | informational and text | Verbosity and whether logs are text or JSON. |

Validation rejects an empty listen address or paths, a public base URL that does not parse, TLS configuration that is incomplete, non-positive timeouts, a sweep or scan interval below a sane floor, a development bind that is not loopback, the presence of encrypted secrets without a master key, and any unrecognised `CAIRN_*` variable.
