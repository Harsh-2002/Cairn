# Bucket replication

> Part of the Cairn reference docs. The section numbers below are stable identifiers used throughout the code and docs; see the index in [`CLAUDE.md`](./CLAUDE.md) and [`../CLAUDE.md`](../CLAUDE.md).

## 20. Bucket replication

### 20.1 Purpose and the consistency model

Bucket replication is Cairn's answer to cross-host and cross-site redundancy, given that clustering is out of scope. It asynchronously copies the objects of a source bucket to a destination bucket on another Cairn deployment or any S3-compatible endpoint, so that a second site holds a current copy that can serve reads or take over on failure. It is eventually consistent with observable lag rather than synchronous, which is the deliberate trade that keeps the write path fast and the deployments independent: a write completes locally and is acknowledged as soon as it is durable on the source, and replication catches the destination up shortly after. This matches the spirit of S3 cross-region replication and is sufficient for redundancy and geo-distribution without the cost and fragility of synchronous cross-site writes.

**Affected versions.** Before release X, a source bucket whose objects were server-side encrypted — by client SSE-S3/SSE-KMS or by `CAIRN_ENCRYPT_AT_REST` — did **not** replicate correctly: the worker read the source blob with no data key and shipped raw ciphertext (§20.4). Depending on whether the object carried a supplementary checksum, the mirror either never received it or received right-sized garbage that answers `200`. Upgrading corrects new writes and heals nothing already on the mirror; detection and repair are §20.5 and `docs/operations.md` 8.7.

### 20.2 Configuration

A bucket's replication configuration names an identity to act as and a list of rules, each with an identifier, a status, a priority that orders overlapping rules, an optional filter by prefix or tags, a destination giving the target endpoint and bucket and the credentials to use there, a setting for whether delete markers are replicated, and a setting for whether existing objects are backfilled when the rule is created in addition to replicating new writes. The destination credentials are stored encrypted at rest under the master key, like the SigV4 secrets, since they grant write access to another system. Following the established model for single-node S3 deployments, a destination is configured per source bucket as a named **remote target** — its endpoint, region, destination bucket, and credentials — identified by a generated resource name (an ARN); a rule references that target by its ARN, and an operator adds a target through the management API, the web console, or the CLI rather than through server configuration. A target to an `https://` endpoint carries its own TLS-trust setting: a **CA certificate** (PEM) to trust when the destination presents a private or self-signed certificate, or — for testing only — a flag to **skip certificate verification**; the two are mutually exclusive, and with neither set the public root anchors are used. These are settable per target through the API and the console (and through the `CAIRN_REPLICATION_TARGETS` env form), so a replica link over a self-signed deployment is configured without touching the system trust store. The destination authenticates the inbound replicated writes with ordinary request signing, so the recommended practice is to create a **dedicated replication user** on the destination whose policy grants exactly the replication actions (`ReplicateObject`, `ReplicateDelete`) on the destination bucket, rather than sharing administrator credentials; this keeps the replication link mutually authenticated even on a plaintext interface. Replication requires the source bucket to have versioning enabled, because a stable per-version identity is what makes replication well-defined and idempotent, so the configuration is rejected unless versioning is enabled, and enabling replication is the operator's signal that this bucket's writes should propagate. A bucket may carry several enabled rules pointing at distinct remote targets. At enqueue **every** enabled rule whose filter matches the written key by prefix and tags is selected, de-duplicated by destination target — when several rules name the same target the highest-priority one wins (ties fall back to document order) — and **one durable outbox entry is enqueued per distinct target**, each stamped with that target's ARN. So a single object **fans out** to every target whose rules select it (true 1→N replication); keyspace-partitioned rules — disjoint prefixes or tags routed to disjoint targets — are simply the degenerate one-entry-per-object case of the same mechanism. Because each entry's ARN is fixed at enqueue time, a later rule edit never misroutes work already queued. Cascading is **not** performed: an object that arrived via replication is marked a replica and enqueues no entries at all (§20.4), so it is never re-fanned-out or forwarded onward.

### 20.3 The durable outbox

Replication is driven by a durable outbox in the metadata store rather than by in-memory state, so that it survives restarts and gives at-least-once delivery with idempotent application. When a write to a replication-enabled bucket commits, and the written version matches a replication rule's filter, an outbox entry is enqueued recording the bucket, key, and version, the operation which is an object creation or a delete-marker propagation, the rule it belongs to, the attempt count, the next attempt time, the status, and the last error. The enqueue is part of the same logical commit as the write, so a committed write is guaranteed to have its replication intent recorded and no write is silently left unreplicated. The outbox is the single source of what remains to be replicated and what has been.

### 20.4 The worker pool and the sink

A pool of replication workers consumes the outbox. A worker claims a batch of entries that are due, respecting **per-key, per-target ordering** so that a key's versions replicate in the order they were written **to each target independently** (a slow or failing target never holds up a healthy one for the same key) and respecting rule priority, and for each entry it drives the destination through the replication sink interface, which is an S3 client capable of putting an object with its content type, user metadata, tags, and ACL, and of propagating a deletion or delete marker. The sink ships the object's **ACL** when the source version carries one, as a base64-encoded JSON header (`x-amz-meta-cairn-replica-acl`), so the destination version reproduces the source's grants. The destination applies it **fail-open**: a malformed or undecodable header is ignored with a warning rather than rejected, because a 4xx is classified terminal and would permanently stall that key's outbox — a bad ACL must degrade to "default ownership", never block replication. It is **dropped** when the destination bucket's ownership mode is `BucketOwnerEnforced` (object ACLs are disabled there, exactly as a client `x-amz-acl` is), and it is gated on the same administrator-role replica classification as the rest of the inbound replica path, so a non-replica write can never apply one. Block-Public-Access is enforced at read time on the destination, so a replicated public-read ACL is subject to the destination's own BPA exactly like a client-set one — no separate write-time check is needed. The delete-marker path carries no ACL. On success the worker marks the entry completed and records on the object version that it has been replicated. Failures are classified by their cause. A failure specific to one object — for example a momentary inability to read the source body — is *retryable*: the worker increments the attempt count and reschedules with capped exponential backoff, and once a configured maximum number of attempts is exhausted the entry is marked terminally failed and surfaced for operator attention rather than retried forever. A failure that means the *destination is unavailable* — a connection failure, a request timeout, a `5xx`, or a throttle (`408`/`429`) — is handled differently: it is rescheduled at a bounded cadence **without consuming the attempt budget**, so a target that is down for an extended period keeps its queued work and resumes automatically when it returns, instead of exhausting to a terminal failure that would need a manual retry. This is what makes node-crash recovery automatic — the entries a peer was owed simply stay pending while it is down (visible as growing lag) and ship when it comes back. (A genuinely-removed target, one no rule resolves to a sink any longer, still terminates via the no-sink path, which *does* consume the budget, so a decommissioned destination's queue does not grow without bound.) Per-key, per-target ordering holds across these states with one deliberate exception: a *terminally-failed* predecessor does not block later versions of the same key+target forever — once it is settled (it will not ship without an operator retry), newer versions proceed, so a single permanent failure cannot silently freeze a key's replication. The destination may then hold a newer version while an older one is permanently missing, which is the documented best-effort consequence; the alternative — a silent indefinite head-of-line stall — is worse. A worker that defers a version to preserve ordering releases its claim immediately (rather than holding the 300 s lease) so the deferred version ships within seconds of its predecessor clearing. A node that crashes mid-drain leaves entries it had *claimed* leased for up to 300 s; to avoid waiting that out, on startup the node releases every `claimed` entry back to `pending` (a freshly-started process holds no claims, so each is an orphan), so it resumes the objects it was actively shipping immediately. **The source version id is preserved end-to-end**: the sink carries it on the wire and the destination stores the replica under that exact id rather than minting a new one, so a logical version has the *same* identity on every node (matching S3 cross-region replication) and re-delivery is an idempotent upsert on `(bucket, key, version_id)` — at-least-once delivery therefore never produces a duplicate version. Because version ids are time-ordered, a replicated version is the destination's *latest* only when its id is the maximum for the key, so a re-delivered or out-of-order older version slots into history without demoting a newer one; under concurrent writes on different nodes this makes the mesh converge **deterministically** (every node ends with the same set of versions and the same latest). Objects that arrived at a bucket through replication are marked as replicas so they are not themselves re-replicated, which prevents loops when two deployments replicate to each other; **cascading is not performed** — a replica enqueues nothing, so an object reaches a node only via a *direct* rule from its origin. A full mesh (every node holding every object) therefore requires an explicit rule from each node to each other node; this is a deliberate trade that makes loop prevention unconditional.

**A server-side-encrypted source version replicates as plaintext, and that has consequences.** An encrypted version's bytes on disk are a block container sealed under a per-object data key which is itself sealed under *this node's* master key; the destination has a different master key and could not open them. The worker therefore resolves the version's data key from its stored SSE descriptor before reading a byte, reads the body through that key, and ships the **logical (plaintext) bytes**, which the destination re-encrypts under its own settings. Before this was implemented the worker read the blob with no key at all and shipped the raw ciphertext, truncated to the logical length: a single-part version was caught by the destination's MD5 check and refused `400 BadDigest` — terminal, so it **never replicated at all** — while a multipart-completed version carries a composite ETag the destination cannot verify, so the ciphertext was accepted and the replica was right-sized, `200`-answering garbage. Failure to resolve the key fails **closed**: nothing is read and nothing is shipped. A key id that is merely not on the ring right now (a rotation window) is treated as *unavailable* — rescheduled without consuming the attempt budget, so a rotation cannot stamp a whole bucket terminally failed — and is reported as a **source-key** condition naming this node's master ring, never as "target unavailable", because the destination is not implicated and an operator must not be sent to the wrong system. Because that path never consumes the attempt budget, a key id removed **permanently** would retry forever and never surface as a failure, so each such reschedule is counted into `cairn_replication_dek_resolve_failed_total`; a sustained non-zero rate there is the signal that objects are silently never shipping. A tampered envelope or an unparseable descriptor can never succeed and is terminal at once.

Shipping plaintext means the transport now carries data a client may have explicitly asked to have encrypted. When the destination endpoint is a plaintext `http://` URL, an object that was encrypted because the **client** requested SSE (SSE-S3 or SSE-KMS) is therefore **refused** per object unless the operator sets `CAIRN_REPLICATION_ALLOW_PLAINTEXT_SSE_OVER_HTTP` (Section 28). The refusal is classified *unavailable*, not terminal: it is an operator-fixable configuration condition, so the object is rescheduled without consuming its attempt budget and ships by itself the moment the endpoint becomes `https://` or the opt-in is set, and the recorded reason names the real cause (an encrypted source object and a plaintext endpoint). Objects that carry only transparent **at-rest** encryption (`CAIRN_ENCRYPT_AT_REST`) are exempt: that is an operator storage property rather than a contract the client asked for, so replicating one over `http://` is no worse than replicating an unencrypted object, and gating it would break existing plaintext-endpoint deployments for no security gain. Enabling the opt-in puts decrypted bodies on an unencrypted, unauthenticated link and should be done only where that link is trusted by other means; it logs a loud startup warning.

### 20.5 Status, observability, and operations

The replication status of each object version is queryable, taking the values pending, completed, failed, or replica, so an operator or the management web console can see whether a given object has reached its destination. The engine exposes metrics for the replication lag measured as the age of the oldest pending entry, the queue depth, the count of completed and of failed transfers, the bytes replicated, and per-destination pending/failed gauges, so that replication health is visible and a growing lag or a rising failure count is alertable. Because lag and queue depth count only *pending* work — they fall to zero once a backlog has exhausted to terminal failure — the engine also exposes an `unreplicated` gauge (pending + in-flight + terminally-failed) that is non-zero whenever any object is owed or stuck, so a dashboard never reads "healthy" while objects are permanently un-shipped. The durable outbox is a bounded *work queue*, not a permanent per-object ledger: completed and terminally-failed rows are reclaimed by a background prune past a retention horizon (so the table does not grow one row per replicated object forever, and stale failures auto-clear), while pending and claimed entries — outstanding work — are never pruned. Failed entries can be retried on demand through the management API after the underlying cause is fixed. A failure to unseal an encrypted source version's data key is counted separately as `cairn_replication_dek_resolve_failed_total`, because it is rescheduled as *unavailable* and therefore never appears in the failed count — it is the only signal that objects are stalled on local key material rather than on the destination. Replication is a **one-way asynchronous mirror**, not synchronous high availability: a destination is a read-only copy for redundancy and geo-distribution, there is no automated promote-to-primary, and the recovery-point objective equals the replication lag at the moment a primary is lost. The operator-facing disaster-recovery procedure — promoting a destination by re-pointing clients, and the no-data-loss guarantees that hold — is the `docs/disaster-recovery.md` runbook. Existing-object backfill (resync) is triggered explicitly through the management API for a bucket whose rule enables existing-object replication: a bounded background pass pages the bucket's current versions and idempotently enqueues an outbox entry for each one a backfill-enabled rule selects, so the existing objects converge to the destination through the same machinery and metrics as new writes, and re-running a resync is safe because the deterministic backfill entry id makes an already-queued or already-completed version a no-op. Because replication writes to another system using stored credentials, it is part of the security surface: the credentials are encrypted at rest, the destination should itself be access-controlled, and the documentation notes that replicating to an untrusted endpoint exports data there.

**The outbox is a work queue, not the incident ledger.** Because completed and terminally-failed rows are pruned past `CAIRN_REPLICATION_RETENTION_SECS` (default 24 hours), the failed-entry listing shows only recent failures and is *dangerously reassuring* about anything older; and the `unreplicated` gauge, being derived from the outbox, does not cover the most dangerous state of all — a version that was shipped **successfully but wrongly**. The durable per-object ledger is the version row's replication status, which is never pruned. The node-local `cairn replication audit` command enumerates it source-side: object versions that carry an SSE descriptor **and** are stamped completed or failed **and** live in a bucket with an enabled rule. It separates the two populations, because they differ entirely in urgency — a *failed* encrypted version is almost certainly **absent** on the mirror (unprotected, but honest), while a *completed* one is **present and suspect** (the replica answers `200` with the right size and may be ciphertext). The suspect predicate additionally requires the version to have been **created before an explicit cutoff**, and that cutoff is supplied by the operator rather than defaulted: only versions written by the pre-fix binary can be damaged, and a version encrypted and replicated *after* the fix is equally encrypted and equally completed, so an unbounded predicate counts every healthy encrypted replica forever and can never reach zero — a signal that cannot converge is not a conservative one, it is a broken one. The cutoff is applied to **two** clocks, not one: the version's creation time, and a replication-owned completion stamp (`object_versions.replicated_at`, written in the same step that marks a version replicated) that a version must lack, or carry from before the cutoff, to remain suspect. Creation time alone cannot converge either, for the mirror-image reason: it is never rewritten, so a damaged version that is force-requeued and then re-ships *correctly* matches the predicate again the moment it is re-stamped completed, and the count returns to its pre-repair value precisely because the repair succeeded. A missing completion stamp — a version never shipped, or one written before the stamp existed — is counted as suspect, so a node upgraded mid-incident over-reports its healthy old versions on the first pass and resolves that as they are re-shipped; an over-report costs a wasted re-ship, while an under-report leaves a garbage replica nobody goes looking for. The same enumeration publishes `cairn_replication_encrypted_suspect_versions`, `cairn_replication_encrypted_absent_versions`, `cairn_replication_encrypted_repair_pending_versions`, and `cairn_replication_encrypted_non_current_suspect_versions` (plus `cairn_replication_audit_pass_seconds`, the wall-clock duration of each audit pass), recomputed on a slow **six-hour background cadence** rather than per scrape: the replication-status column carries no index, so a metrics scrape must never trigger the walk. The background pass is **opt-in and off by default** (`CAIRN_REPLICATION_AUDIT_BEFORE`, Section 28): unset, the loop is never started, no gauge is emitted and no warning fires, because a permanently-firing alarm shipped as a default is worse than no alarm. The third gauge exists because the other two cannot express convergence on their own: a forced requeue moves the damaged rows to *pending*, which the suspect predicate excludes, so the suspect count falls to zero the instant the repair is queued — before a byte re-ships — and then climbs as entries complete. Repair is finished when the repair-pending count is zero **and** the suspect count has fallen to the non-current floor — the fourth gauge — rather than to zero: a forced requeue moves a version to *pending* only when something can actually ship it, meaning the version is current (the backfill re-enqueues current objects) or it still has an outbox row, so a superseded version whose row the retention sweep already pruned stays counted as a suspect instead of being marked as queued work that no queue holds. That distinction is what makes the repair-pending series convergent at all; the alternative marks rows *pending* forever and leaves any alert on it permanently firing. Those non-current versions are unrepairable in place — clearing them means rebuilding the destination bucket. The pass's cost is genuinely two-sided: buckets without an enabled rule are skipped before any version is read, so a store that does not replicate costs almost nothing, but a bucket that *does* replicate costs a full version listing **plus one point query per listed version**, because the listing page carries no SSE descriptor — on a large replicated bucket that per-version read is the dominant cost, and it is the second reason the cadence is slow and the loop opt-in. Narrowing the suspect set by comparing against the remote is **not possible** and must not be attempted: the worker requested the *plaintext* byte range, so a corrupt replica is exactly the right length, and a multipart source lands as a single-part PUT, so the ETag differs even for a correct replica. The only conclusive check is a full byte comparison against the re-derived source plaintext, which `--verify` performs at the cost of transferring every suspect object.

**Repair re-ships unconditionally.** Because nothing can be diffed, remediation does not try: it re-enqueues the suspect versions and lets the (now correct) worker overwrite the destination. A first resync already achieves this for versions whose outbox row has been pruned — the worker deliberately does *not* skip a version stamped completed, since under 1→N fan-out that would starve every target but the first. A **second** pass inside the retention window, however, is silently a no-op, because the deterministic backfill entry id still has a row and the enqueue is idempotent. `POST /buckets/{name}/replication/resync?force=true` closes that gap: it first flips the bucket's terminal outbox rows back to pending with the attempt budget reset and resets their version-row stamps, then runs the ordinary backfill. The requeue is scoped to the **key**, not the individual version: unless `only_encrypted=false` widens it to the whole bucket, it selects every key that has at least one encrypted version and re-queues *all* of that key's terminal entries, including later plaintext versions and delete markers. Filtering per version would be a correctness defect rather than an optimisation — re-shipping only the encrypted version puts it at the destination *after* newer versions already there, so the mirror's current object reverts to an older one, and a delete marker that is never re-shipped resurrects a deleted object; with every version of the key queued the engine's write-ordering guard restores the original order. The requeue is also **paged, and the page unit is the key rather than the row**: an unbounded update would hold one group-commit transaction across a full-table scan and stall every write on the node, so each transaction covers a bounded number of distinct keys, moves every terminal row of every key in that page together, and returns that page's last key as a forward cursor the caller threads back so no pass rescans what an earlier one drained. Paging by row instead would reintroduce the ordering defect the key scope exists to prevent, because rows would be served in index order — every completed row ahead of every failed one — and a key whose older version failed and whose newer version completed would have the newer one requeued pages ahead of the older, letting the heartbeat ship them out of order. The one deliberately unbounded case is a single key with an enormous version history, whose rows are never split across transactions. If the paging ever exhausts its batch budget with keys still unrequeued, the node logs a loud non-convergence line carrying the resume cursor and records it in the activity feed, because a half-done repair reported as done is the one failure this mechanism must never have. A forced resync on a very large bucket remains write-heavy in total volume, which the operator runbook states. Three constraints bound what this can repair, and all three are stated in the operator runbook (`docs/operations.md` 8.7) because each silently wastes a pass: the backfill only runs for rules that enable existing-object replication; it enumerates **current versions only**, so a non-current version that replicated corrupt is not repaired at all and full version-history fidelity requires rebuilding the destination bucket; and because repair ships **plaintext**, a client-encrypted object bound for an `http://` endpoint is refused by the confidentiality gate and rescheduled indefinitely rather than failing, so the endpoint must be `https://` (or the opt-in set) before a repair will make progress.

### 20.6 Event notifications (webhooks)

Event notifications reuse the replication outbox machinery to deliver object-event callbacks. They are deliberately **webhook-native** rather than the S3 SNS/SQS/Lambda shape — those target AWS ARNs Cairn has no equivalent for — so a bucket's notification configuration is a list of webhook endpoints set through the management API (`PUT /api/v1/buckets/{name}/notifications`), and the S3 `?notification` subresource stays not-implemented. Each endpoint carries a URL, a set of event selectors (`s3:ObjectCreated:*`, `s3:ObjectRemoved:*`, or a specific event such as `s3:ObjectCreated:Put`), optional object-key prefix/suffix filters, and an optional HMAC secret.

When an object event fires — a PUT, a multipart completion, a copy, a permanent delete, or a delete-marker creation — the protocol layer reads the bucket's (cached) notification config, matches each endpoint by event and prefix/suffix, renders the standard S3 event-record JSON (so existing webhook consumers parse it unchanged), and enqueues one delivery row per matching endpoint into a dedicated `events_outbox` table. The enqueue is best-effort at-least-once, matching S3's own event-delivery contract: it rides just after the object commit rather than inside its transaction, so a crash in the gap drops the notification, never the object, and a notification failure can never fail the originating object operation. A background worker — mirroring the replication drain loop, claiming a batch under a lease so two workers never deliver the same row — POSTs each event to its endpoint, signing the body as `X-Cairn-Signature: sha256=<hmac>` when a secret is configured, and classifies the response: a 2xx completes the row, a 5xx/408/429 or a transport error reschedules it with exponential backoff, and any other 4xx is terminal. An endpoint whose subscription was removed since enqueue drops its pending rows rather than retrying forever. The claim is a cheap indexed query, so a bucket with no notifications configured costs the worker nothing. Because deliveries leave the node carrying object metadata (and optionally an HMAC of the body) to an external URL, an endpoint should be a trusted, access-controlled receiver — the same security posture as a replication destination.
