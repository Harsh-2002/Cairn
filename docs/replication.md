## 20. Bucket replication

### 20.1 Purpose and the consistency model

Bucket replication is Cairn's answer to cross-host and cross-site redundancy, given that clustering is out of scope. It asynchronously copies the objects of a source bucket to a destination bucket on another Cairn deployment or any S3-compatible endpoint, so that a second site holds a current copy that can serve reads or take over on failure. It is eventually consistent with observable lag rather than synchronous, which is the deliberate trade that keeps the write path fast and the deployments independent: a write completes locally and is acknowledged as soon as it is durable on the source, and replication catches the destination up shortly after. This matches the spirit of S3 cross-region replication and is sufficient for redundancy and geo-distribution without the cost and fragility of synchronous cross-site writes.

### 20.2 Configuration

A bucket's replication configuration names an identity to act as and a list of rules, each with an identifier, a status, a priority that orders overlapping rules, an optional filter by prefix or tags, a destination giving the target endpoint and bucket and the credentials to use there, a setting for whether delete markers are replicated, and a setting for whether existing objects are backfilled when the rule is created in addition to replicating new writes. The destination credentials are stored encrypted at rest under the master key, like the SigV4 secrets, since they grant write access to another system. Following the established model for single-node S3 deployments, a destination is configured per source bucket as a named **remote target** — its endpoint, region, destination bucket, and credentials — identified by a generated resource name (an ARN); a rule references that target by its ARN, and an operator adds a target through the management API, the UI, or the CLI rather than through server configuration. The destination authenticates the inbound replicated writes with ordinary request signing, so the recommended practice is to create a **dedicated replication user** on the destination whose policy grants exactly the replication actions (`ReplicateObject`, `ReplicateDelete`) on the destination bucket, rather than sharing administrator credentials; this keeps the replication link mutually authenticated even on a plaintext interface. Replication requires the source bucket to have versioning enabled, because a stable per-version identity is what makes replication well-defined and idempotent, so the configuration is rejected unless versioning is enabled, and enabling replication is the operator's signal that this bucket's writes should propagate. A bucket may carry several enabled rules pointing at distinct remote targets. At enqueue **every** enabled rule whose filter matches the written key by prefix and tags is selected, de-duplicated by destination target — when several rules name the same target the highest-priority one wins (ties fall back to document order) — and **one durable outbox entry is enqueued per distinct target**, each stamped with that target's ARN. So a single object **fans out** to every target whose rules select it (true 1→N replication); keyspace-partitioned rules — disjoint prefixes or tags routed to disjoint targets — are simply the degenerate one-entry-per-object case of the same mechanism. Because each entry's ARN is fixed at enqueue time, a later rule edit never misroutes work already queued. Cascading is **not** performed: an object that arrived via replication is marked a replica and enqueues no entries at all (§20.4), so it is never re-fanned-out or forwarded onward.

### 20.3 The durable outbox

Replication is driven by a durable outbox in the metadata store rather than by in-memory state, so that it survives restarts and gives at-least-once delivery with idempotent application. When a write to a replication-enabled bucket commits, and the written version matches a replication rule's filter, an outbox entry is enqueued recording the bucket, key, and version, the operation which is an object creation or a delete-marker propagation, the rule it belongs to, the attempt count, the next attempt time, the status, and the last error. The enqueue is part of the same logical commit as the write, so a committed write is guaranteed to have its replication intent recorded and no write is silently left unreplicated. The outbox is the single source of what remains to be replicated and what has been.

### 20.4 The worker pool and the sink

A pool of replication workers consumes the outbox. A worker claims a batch of entries that are due, respecting **per-key, per-target ordering** so that a key's versions replicate in the order they were written **to each target independently** (a slow or failing target never holds up a healthy one for the same key) and respecting rule priority, and for each entry it drives the destination through the replication sink interface, which is an S3 client capable of putting an object with its content type, user metadata, and tags, and of propagating a deletion or delete marker. The sink does not currently ship the object's ACL; ACL replication is future work. On success the worker marks the entry completed and records on the object version that it has been replicated. Failures are classified by their cause. A failure specific to one object — for example a momentary inability to read the source body — is *retryable*: the worker increments the attempt count and reschedules with capped exponential backoff, and once a configured maximum number of attempts is exhausted the entry is marked terminally failed and surfaced for operator attention rather than retried forever. A failure that means the *destination is unavailable* — a connection failure, a request timeout, a `5xx`, or a throttle (`408`/`429`) — is handled differently: it is rescheduled at a bounded cadence **without consuming the attempt budget**, so a target that is down for an extended period keeps its queued work and resumes automatically when it returns, instead of exhausting to a terminal failure that would need a manual retry. This is what makes node-crash recovery automatic — the entries a peer was owed simply stay pending while it is down (visible as growing lag) and ship when it comes back. (A genuinely-removed target, one no rule resolves to a sink any longer, still terminates via the no-sink path, which *does* consume the budget, so a decommissioned destination's queue does not grow without bound.) Per-key, per-target ordering holds across these states with one deliberate exception: a *terminally-failed* predecessor does not block later versions of the same key+target forever — once it is settled (it will not ship without an operator retry), newer versions proceed, so a single permanent failure cannot silently freeze a key's replication. The destination may then hold a newer version while an older one is permanently missing, which is the documented best-effort consequence; the alternative — a silent indefinite head-of-line stall — is worse. A worker that defers a version to preserve ordering releases its claim immediately (rather than holding the 300 s lease) so the deferred version ships within seconds of its predecessor clearing. A node that crashes mid-drain leaves entries it had *claimed* leased for up to 300 s; to avoid waiting that out, on startup the node releases every `claimed` entry back to `pending` (a freshly-started process holds no claims, so each is an orphan), so it resumes the objects it was actively shipping immediately. **The source version id is preserved end-to-end**: the sink carries it on the wire and the destination stores the replica under that exact id rather than minting a new one, so a logical version has the *same* identity on every node (matching S3 cross-region replication) and re-delivery is an idempotent upsert on `(bucket, key, version_id)` — at-least-once delivery therefore never produces a duplicate version. Because version ids are time-ordered, a replicated version is the destination's *latest* only when its id is the maximum for the key, so a re-delivered or out-of-order older version slots into history without demoting a newer one; under concurrent writes on different nodes this makes the mesh converge **deterministically** (every node ends with the same set of versions and the same latest). Objects that arrived at a bucket through replication are marked as replicas so they are not themselves re-replicated, which prevents loops when two deployments replicate to each other; **cascading is not performed** — a replica enqueues nothing, so an object reaches a node only via a *direct* rule from its origin. A full mesh (every node holding every object) therefore requires an explicit rule from each node to each other node; this is a deliberate trade that makes loop prevention unconditional.

### 20.5 Status, observability, and operations

The replication status of each object version is queryable, taking the values pending, completed, failed, or replica, so an operator or the management UI can see whether a given object has reached its destination. The engine exposes metrics for the replication lag measured as the age of the oldest pending entry, the queue depth, the count of completed and of failed transfers, the bytes replicated, and per-destination pending/failed gauges, so that replication health is visible and a growing lag or a rising failure count is alertable. Because lag and queue depth count only *pending* work — they fall to zero once a backlog has exhausted to terminal failure — the engine also exposes an `unreplicated` gauge (pending + in-flight + terminally-failed) that is non-zero whenever any object is owed or stuck, so a dashboard never reads "healthy" while objects are permanently un-shipped. The durable outbox is a bounded *work queue*, not a permanent per-object ledger: completed and terminally-failed rows are reclaimed by a background prune past a retention horizon (so the table does not grow one row per replicated object forever, and stale failures auto-clear), while pending and claimed entries — outstanding work — are never pruned. Failed entries can be retried on demand through the management API after the underlying cause is fixed. Existing-object backfill (resync) is triggered explicitly through the management API for a bucket whose rule enables existing-object replication: a bounded background pass pages the bucket's current versions and idempotently enqueues an outbox entry for each one a backfill-enabled rule selects, so the existing objects converge to the destination through the same machinery and metrics as new writes, and re-running a resync is safe because the deterministic backfill entry id makes an already-queued or already-completed version a no-op. Because replication writes to another system using stored credentials, it is part of the security surface: the credentials are encrypted at rest, the destination should itself be access-controlled, and the documentation notes that replicating to an untrusted endpoint exports data there.

### 20.6 Event notifications (webhooks)

Event notifications reuse the replication outbox machinery to deliver object-event callbacks. They are deliberately **webhook-native** rather than the S3 SNS/SQS/Lambda shape — those target AWS ARNs Cairn has no equivalent for — so a bucket's notification configuration is a list of webhook endpoints set through the management API (`PUT /api/v1/buckets/{name}/notifications`), and the S3 `?notification` subresource stays not-implemented. Each endpoint carries a URL, a set of event selectors (`s3:ObjectCreated:*`, `s3:ObjectRemoved:*`, or a specific event such as `s3:ObjectCreated:Put`), optional object-key prefix/suffix filters, and an optional HMAC secret.

When an object event fires — a PUT, a multipart completion, a copy, a permanent delete, or a delete-marker creation — the protocol layer reads the bucket's (cached) notification config, matches each endpoint by event and prefix/suffix, renders the standard S3 event-record JSON (so existing webhook consumers parse it unchanged), and enqueues one delivery row per matching endpoint into a dedicated `events_outbox` table. The enqueue is best-effort at-least-once, matching S3's own event-delivery contract: it rides just after the object commit rather than inside its transaction, so a crash in the gap drops the notification, never the object, and a notification failure can never fail the originating object operation. A background worker — mirroring the replication drain loop, claiming a batch under a lease so two workers never deliver the same row — POSTs each event to its endpoint, signing the body as `X-Cairn-Signature: sha256=<hmac>` when a secret is configured, and classifies the response: a 2xx completes the row, a 5xx/408/429 or a transport error reschedules it with exponential backoff, and any other 4xx is terminal. An endpoint whose subscription was removed since enqueue drops its pending rows rather than retrying forever. The claim is a cheap indexed query, so a bucket with no notifications configured costs the worker nothing. Because deliveries leave the node carrying object metadata (and optionally an HMAC of the body) to an external URL, an endpoint should be a trusted, access-controlled receiver — the same security posture as a replication destination.
