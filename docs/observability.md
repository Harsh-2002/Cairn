# Observability and audit

> Part of the Cairn reference docs. The section numbers below are stable identifiers used throughout the code and docs; see the index in [`CLAUDE.md`](./CLAUDE.md) and [`../CLAUDE.md`](../CLAUDE.md).

## 26. Observability and audit

### 26.1 Tracing and logging

The system is instrumented with structured tracing throughout, with one span per request that carries the method, the route, the bucket and key, the principal, the request identifier, the resulting status, and the duration, and with structured logs that can be emitted as human-readable text or as machine-readable JSON depending on configuration, filtered by a configurable level. Errors are logged at the boundary with their request identifier so they correlate with the client-visible identifier and with the trace. This is the baseline of being able to see what the server is doing, which Cairn provides in full.

### 26.2 Metrics

A metrics endpoint in the widely-supported text exposition format publishes the series an operator needs to run Cairn in production (F-18). The set includes request counts and a latency histogram labelled by route and method and status; bytes received and sent; gauges for object, bucket, and version counts and the logical and physical stored-byte totals refreshed from the store and on mutation; counters for metadata cache hits and misses; the depth of the metadata writer's queue, which is the key early-warning signal for the single-writer write-rate ceiling so an operator can see write saturation before it becomes latency; the write-ahead-log checkpoint runs, the count of checkpoints that returned busy against a live reader, and the log size; the replication queue depth, lag, completed and failed counts, and bytes replicated; the count of replication attempts rescheduled because the *source* object's data key could not be unsealed on this node's master ring (`cairn_replication_dek_resolve_failed_total`, which is the only durable signal for that condition — such reschedules never consume the attempt budget and so never appear in the failed count); the count of blob reads refused because an encrypted blob was read without a data key (`cairn_blob_encrypted_without_key_total`, the fail-closed guard, alertable where a log line would not be); the count of object versions that are encrypted, stamped terminally replicated, and created before an operator-supplied cutoff (`cairn_replication_encrypted_suspect_versions` for those the mirror accepted — present and possibly ciphertext — `cairn_replication_encrypted_absent_versions` for those it refused, and `cairn_replication_encrypted_repair_pending_versions` for those a forced repair has re-queued but not yet re-shipped, alongside `cairn_replication_encrypted_non_current_suspect_versions` for the part of the suspect population no repair command reaches — the resync backfill enumerates current versions only, and a forced requeue deliberately does not mark a superseded version whose outbox row has been pruned as queued, since nothing would ever ship it and the repair-pending series could then never return to zero), which is the only signal for the pre-release-X plaintext-seam damage because those versions read as *completed* and are therefore invisible to `cairn_replication_unreplicated`; the cutoff is what makes the series convergent, since a version encrypted and replicated after the fix is equally encrypted and equally completed and would otherwise be counted forever, and it is applied to two clocks rather than one — a version's creation time and its `replicated_at` completion stamp (schema v23, written by the replication engine in the same step that marks a version replicated) — because creation time alone can never let a *repaired* version out of the population, its creation time being immutable, so the gauge would return to its pre-repair value precisely because the repair succeeded; a version whose completion stamp is missing (never shipped, or written before that migration) is counted as suspect, which means a node upgraded mid-incident over-reports its healthy old versions on the first pass and resolves that as they are re-shipped, an over-report costing one wasted re-ship where an under-report would leave a garbage replica nobody looks for; the repair-pending series is what makes the whole thing readable, since a requeue moves damaged rows into a state the suspect predicate excludes and the suspect count therefore falls to zero before any byte re-ships — the repair is complete when the repair-pending series is zero and the suspect series has fallen to the non-current floor, which is why that floor is published as its own series rather than left for an operator to infer; these are **opt-in and emitted only when the cutoff is configured** (`CAIRN_REPLICATION_AUDIT_BEFORE`), and are recomputed on a slow six-hour background pass rather than per scrape, since the underlying replication-status column carries no index, the pass costs one point query per version in a replicating bucket, and a scrape must never trigger a table walk (`cairn_replication_audit_pass_seconds` reports what the pass costs); and the logical-versus-physical byte totals that express the compression ratio. These series make every subsystem specified in this document observable, which is what turns the design into something operable.

### 26.3 Audit log

Mutating actions across both the S3 and management surfaces are recorded in an audit log with the actor, the action, the resource, and the salient attributes, retained in the metadata store and surfaced through the management API and web console. This serves both the operational need to see recent activity and the security need to have a record of who changed or accessed what, and it is distinct from the operational metrics in that it is per-event and attributable rather than aggregate.

### 26.4 Health, readiness, and optional scrub

The system exposes a liveness endpoint that succeeds whenever the process is serving and a readiness endpoint that succeeds only once migrations have applied, reconciliation has completed or been deliberately deferred, and the writer and read pool are responsive, so that an orchestrator does not route traffic to a process that is not yet ready. As an opt-in integrity facility, a background scrub (`CAIRN_SCRUB_INTERVAL_SECS`, **off by default** — with no interval configured no pass ever runs) re-reads stored blobs on the configured schedule and verifies them against their recorded ETag, reporting a detected mismatch as `cairn_scrub_corruption_total` (with `cairn_scrub_objects_total` and `cairn_scrub_pass_seconds` for progress) rather than serving the corrupted byte. **Encrypted versions are verified like any other**: the pass unseals each version's data key and re-reads through it, so transparent at-rest, SSE-S3 and SSE-KMS objects are covered — the read authenticates every AES-GCM block and the recomputed plaintext MD5 is compared to the stored ETag, which is a plaintext digest under encryption too.

Because a scrub that quietly verifies nothing is indistinguishable from a clean store, what a pass cannot verify is **counted and emitted, never dropped**: `cairn_scrub_skipped_total{reason}` breaks the residue down by `key_unavailable` (the version's key is not on the ring right now — a rotation window; skipped and retried next pass, deliberately **not** reported as corruption, since a false bit-rot alarm on a healthy store is its own incident), `io_error` (a transient filesystem failure — fd exhaustion, a full disk — skipped for the same reason and retried; a *missing* blob file, by contrast, is reported as `cairn_scrub_corruption_total{kind="missing_blob"}`), `composite_etag` (a multipart `{md5}-{n}` ETag is a hash of hashes, so the object is read and authenticated but its content hash is not compared — composite verification is out of scope), `no_blob`, `delete_marker`, and `metadata_unavailable`. A metadata listing that fails is counted as `cairn_scrub_enumeration_errors_total{stage}` (the pass still emits its log line rather than aborting silently, but covers less than the whole store). The invariant, which the pass log line (`scanned`, `skipped`, `corrupt`, `walked` and the per-reason fields) makes checkable, is that **`cairn_scrub_objects_total` + `cairn_scrub_skipped_total` account for every version the pass walked**; an operator alert should therefore look at both, since `scanned=0` with a non-zero `skipped` means coverage was lost, not that the store is clean. Note the scrub now re-reads and *decrypts* every version, so on an encrypted store a pass is CPU- and I/O-heavy in proportion to stored bytes (there is no rate limiter; it takes one blob-read permit at a time so live GETs are not starved) — schedule the interval for quiet periods. The scrub complements the redundancy the operator provides at the storage layer by giving early warning of bit rot.

### 26.5 Request metrics and usage analytics

Distinct from the scrape-only Prometheus series of Section 26.2, which are in-memory counters an external monitoring system samples, Cairn keeps a **stored, queryable time-series of API request activity** so the console can present a self-contained usage-analytics view without an external metrics stack. The two design constraints are that the request hot path must do no database work, and that a rolling month of history must stay bounded. Both are met by aggregating in memory and flushing in batches. Every completed request is classified into an operation name — `GetObject`, `PutObject`, `ListObjects`, the multipart operations, the coarse `Management` bucket for the control plane, and so on — and the request's target bucket and HTTP status class are noted; this tuple, floored to a configurable time window, indexes an in-process sharded counter that the request increments under a microsecond-held lock and nothing more. Alongside the count, each request also folds its **transferred bytes** (received and sent) and its **latency** — both as a running sum and into a small fixed-bucket histogram — into the same accumulated cell, so the hot path stays a single in-memory update. A background task periodically drains the accumulated cells and submits them as a single batched mutation through the same single writer that owns every other write, where they upsert-accumulate into a rollup table keyed by window, operation, bucket, and status class; the same flush prunes rows older than the retention horizon, so the table is self-bounding. Because storage is one row per active tuple per window rather than one row per request, a busy node costs a bounded number of rows per window, and the flush amortises all of a window's traffic into one transaction regardless of request rate.

The query side answers a range — one day, one week, two weeks, or one month — by reading the rollup since that range's lower bound and **downsampling the timeline into a window chosen for the range** (five minutes for a day, an hour for a week, three hours for a fortnight, six hours for a month), so the returned series has a bounded number of points whatever the underlying row count. Each timeline point carries its requests, errors, bytes, and average latency, and the query also returns the breakdown by operation (with bytes and latency), by bucket, and by status class, plus range-wide totals: request and error counts, bytes moved, the average and a histogram-estimated 95th-percentile latency, the busiest window, and the number of active buckets. From these the console renders a Grafana-style responsive dashboard — stat tiles for volume, throughput, error rate, and latency alongside system storage figures, and time-series and breakdown panels — that reflows from three columns on a desktop to one on a phone. The whole subsystem is gated by configuration and, when disabled, neither accumulates nor spawns its flush, and its window granularity, flush cadence, and retention are operator-tunable (Section 28.2). This is the per-tenant, per-operation visibility an operator needs to understand the shape and cost of their traffic, provided from Cairn's own store rather than requiring a separate observability deployment.
